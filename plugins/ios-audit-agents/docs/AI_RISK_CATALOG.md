# AI Risk Catalog

> **Catalog v0.1.** Stable identifiers for AI-typical risks referenced
> from `QUALITY_FRAMEWORK.md`, used inside every audit output to enable
> aggregation, recurrence tracking, and longitudinal analysis.

## How to read this catalog

Each entry is a hypothesis: "AI-assisted iOS code is more likely than
handwritten code to exhibit this pattern." The catalog is the
plugin's vocabulary for that hypothesis. Whether a given risk recurs
in practice is a question the audit data answers (see `QUALITY_FRAMEWORK.md`
Section 7.3).

### Identifier scheme

```
AI-3.X-NNN
```

- `3.X` — quality dimension from the quality model.
- `NNN` — zero-padded sequential index inside the dimension.

Identifiers are stable. They are never reused once retired. A retired
risk is kept in this document with `status: retired` and a short
rationale.

### Status values

- **`cited`** — at least one explicit, public source describes this
  pattern as a recurring AI deviation.
- **`common knowledge`** — broadly agreed in the iOS community, no
  single canonical source, but consistently observed.
- **`framework heuristic`** — introduced by this catalog as a
  testable hypothesis. Will be promoted, reworded, or retired based
  on accumulated audit data.
- **`retired`** — no longer used. ID kept for traceability.

### Entry shape

Each risk entry includes:

- **ID, title, dimension, status.**
- **Description** — one paragraph stating the risk in plain terms.
- **AI manifestation** — how an AI copilot tends to introduce it.
- **Example** — minimal illustration (no full code, no real secrets).
- **Sources** — citations or pointers, when applicable.

## Dimension 3.1 — Product fit & boundaries

### AI-3.1-001 — Speculative features outside the spec

- **Status:** framework heuristic.
- **Description:** The codebase contains code paths or feature flags
  for capabilities that were never part of the product specification,
  added "just in case" or generated by an AI extrapolating beyond the
  prompt.
- **AI manifestation:** A copilot, asked to "add login", also adds
  password reset, social login, and biometrics, none of which were
  requested or designed.
- **Example:** An unowned `enum AuthProvider { case email, google,
  apple, facebook }` with only `email` actually wired to UI.
- **Sources:** framework heuristic; informed by retrospectives on
  AI-assisted feature work.

## Dimension 3.2 — Architecture & modularity

### AI-3.2-001 — Plausible-looking but inconsistent layering

- **Status:** framework heuristic.
- **Description:** The project advertises a layered architecture
  (e.g. MVVM + Coordinator + Repository) but the layers leak: views
  fetch from network, view models persist directly, repositories
  contain UI logic.
- **AI manifestation:** AI tends to produce names that match a
  pattern even when the responsibilities do not. The structure looks
  correct from a directory listing but breaks under inspection.
- **Example:** A `HomeViewModel` that imports `URLSession` and
  decodes JSON inline.
- **Sources:** framework heuristic; aligned with general guidance on
  layered iOS architectures (Swift by Sundell — architecture topics).

### AI-3.2-002 — Direct cross-layer dependencies

- **Status:** common knowledge.
- **Description:** A higher-level component (typically a view)
  imports a lower-level one (network, persistence) without going
  through an abstraction.
- **AI manifestation:** Asked to "add a button that loads data", AI
  often inlines the network call in `.task { ... }` instead of going
  through a view model or repository.
- **Example:** `import Foundation` + `URLSession.shared.dataTask`
  inside `Button { ... }` action.
- **Sources:** common knowledge in the iOS community; reinforced by
  Swift by Sundell content on separation of concerns.

## Dimension 3.3 — Domain model integrity

### AI-3.3-001 — Stringly-typed identifiers and weak invariants

- **Status:** common knowledge.
- **Description:** Identifiers are raw `String` everywhere; enums are
  replaced with string constants; invariants are not enforced by the
  type system.
- **AI manifestation:** AI gravitates to `String` because it is the
  most generic type that compiles. Asked for "user id", it produces
  `let id: String` instead of a typed `UserID`.
- **Example:** `func loadUser(id: String)` and another
  `func loadOrder(id: String)` accepting the same type.
- **Sources:** common knowledge; reinforced by Swift by Sundell
  (typed identifiers and value semantics).

## Dimension 3.4 — State, concurrency & data races

### AI-3.4-001 — Concurrency band-aids that hide isolation issues

- **Status:** cited / common knowledge.
- **Description:** `@MainActor`, `DispatchQueue.main.async`, or
  `Task { @MainActor in ... }` are sprinkled to silence warnings or
  fix observed crashes, instead of designing actor boundaries.
- **AI manifestation:** Faced with a data race warning, AI tends to
  add the smallest annotation that compiles, frequently the wrong one
  (e.g. `@MainActor` on a property that should not be UI-bound).
- **Example:** A `Repository` class declared `@MainActor` because a
  view used it; now all its work blocks the main thread.
- **Sources:** Donny Wals (Swift Concurrency content);
  Antoine van der Lee (Swift 6 migration tools and articles).

### AI-3.4-002 — Unbounded or uncancelled tasks

- **Status:** common knowledge.
- **Description:** `Task { ... }` blocks created without cancellation,
  long-running observers without lifecycle anchors, leaks on view
  dismissal.
- **AI manifestation:** AI uses `Task { ... }` as a fire-and-forget
  primitive without coupling its lifetime to the view that owns it.
- **Example:** `.onAppear { Task { await poll() } }` with no
  `.task { ... }` and no `Task.isCancelled` check.
- **Sources:** common knowledge; reinforced by Apple Swift Concurrency
  documentation.

## Dimension 3.5 — Reliability & error handling

### AI-3.5-001 — Silent error swallowing

- **Status:** cited.
- **Description:** Errors are caught and ignored, or `try?` is used
  to throw away failures, leaving the user staring at a blank screen.
- **AI manifestation:** AI prefers code that compiles cleanly. The
  shortest path is to swallow the error: `try?`, empty `catch`,
  `if let _ = try? ...`.
- **Example:** `let data = try? Data(contentsOf: url)`, with `data`
  silently `nil` on any error.
- **Sources:** Paul Hudson — *What to fix in AI-generated Swift code*
  (`hudson:281`).

### AI-3.5-002 — Plausible but ineffective retry/fallback logic

- **Status:** framework heuristic.
- **Description:** Retry code that runs once with no backoff,
  fallbacks that do not actually recover, or "offline" branches that
  return placeholder data without the user knowing.
- **AI manifestation:** AI generates the shape of robustness without
  the substance: a `for _ in 0..<3` loop with no jitter, or a `catch`
  that returns `[]` and lets the UI show "no items".
- **Example:** `try await api.fetch()` wrapped in a 3-iteration loop
  with `try? await Task.sleep(for: .seconds(1))` and no jitter.
- **Sources:** framework heuristic.

## Dimension 3.6 — Security & privacy

### AI-3.6-001 — Hardcoded secrets or unsafe storage

- **Status:** common knowledge.
- **Description:** API keys, tokens or passwords appear in source
  code, in `UserDefaults`, in plist files, or in environment-like
  constants.
- **AI manifestation:** AI fills in placeholder secrets to make code
  runnable in isolation; those placeholders survive into production.
- **Example:** `let apiKey = "sk_live_abc123..."` in a `Constants.swift`.
- **Sources:** OWASP MASVS v2.1 — STORAGE; Apple App Review Guidelines
  — Safety/Legal.

### AI-3.6-002 — App Transport Security relaxations

- **Status:** common knowledge.
- **Description:** `NSAllowsArbitraryLoads` set to `true` or broad
  exception domains, often "for development", left in the shipped
  binary.
- **AI manifestation:** AI provides the broadest fix to make a
  request work, instead of adding the specific exception or fixing
  the server-side certificate.
- **Example:** `NSAppTransportSecurity / NSAllowsArbitraryLoads = YES`
  in the production target's `Info.plist`.
- **Sources:** Apple App Transport Security documentation;
  OWASP MASVS v2.1 — NETWORK.

### AI-3.6-003 — Missing or inaccurate privacy manifest

- **Status:** common knowledge.
- **Description:** `PrivacyInfo.xcprivacy` absent, missing required
  reasons for tracked APIs, or declared tracking domains that do not
  match what the app actually contacts.
- **AI manifestation:** AI does not generate `PrivacyInfo.xcprivacy`
  unless asked; it also does not update it when adding new SDKs.
- **Example:** A new analytics SDK is integrated; the privacy manifest
  is unchanged.
- **Sources:** Apple Privacy manifest documentation; App Review
  Guidelines — Privacy.

## Dimension 3.7 — Observability & telemetry

### AI-3.7-001 — `print` instead of structured logging

- **Status:** cited.
- **Description:** Logging happens with `print(...)` instead of
  `OSLog` (`Logger`) with subsystem and category, making logs
  unfilterable in Console and Instruments.
- **AI manifestation:** AI defaults to `print` because it always
  works and requires no setup.
- **Example:** `print("user logged in: \(user.email)")`.
- **Sources:** Apple OSLog documentation; common iOS practice.

### AI-3.7-002 — PII in logs

- **Status:** framework heuristic.
- **Description:** Logs include email addresses, user identifiers,
  full payloads, or other personal data without redaction or privacy
  classification.
- **AI manifestation:** AI logs the variable that is most informative
  for debugging, which is often the personal data itself.
- **Example:** `Logger().info("loaded order for \(user.email)")`.
- **Sources:** OWASP MASVS v2.1 — PRIVACY; Apple Privacy guidelines.

## Dimension 3.8 — Testing strategy

### AI-3.8-001 — Tests assert implementation details, not behavior

- **Status:** framework heuristic.
- **Description:** Tests verify that specific functions are called,
  in a specific order, on specific mocks; they break on every
  refactor and prove little about user-facing behavior.
- **AI manifestation:** AI generates tests that mirror the structure
  of the code under test, including its private steps.
- **Example:** A test that asserts `mockRepository.fetchCalledTimes
  == 1` after invoking a view model action, without checking the
  output state.
- **Sources:** framework heuristic; aligned with general testing
  literature.

### AI-3.8-002 — Mock-everything tests with no real assertion

- **Status:** framework heuristic.
- **Description:** All dependencies are mocked, all return values
  hardcoded, and the only assertion is that "nothing crashed".
- **AI manifestation:** AI satisfies a "write a test for X" prompt
  with the smallest scaffold that compiles and runs.
- **Example:** A `func test_loadHome_doesNotCrash()` whose only
  assertion is `#expect(true)`.
- **Sources:** framework heuristic.

## Dimension 3.9 — CI/CD & release engineering

### AI-3.9-001 — Required gates not actually required

- **Status:** common knowledge.
- **Description:** Lint, tests or coverage checks run in CI but are
  not enforced as merge gates; failures are visible but ignored.
- **AI manifestation:** AI scaffolds CI workflows that *run* the
  checks; whether they *block* a merge is a separate configuration
  step that AI rarely sets.
- **Example:** A GitHub Actions workflow that runs `swiftlint` but
  branch protection does not require its status check.
- **Sources:** common knowledge in CI/CD practice.

## Dimension 3.10 — Developer experience (DX)

### AI-3.10-001 — README drift from real bootstrap

- **Status:** framework heuristic.
- **Description:** The setup steps in `README.md` no longer match
  what is needed for a clean machine to build and run the app.
- **AI manifestation:** AI updates code without updating
  documentation; documentation updates often produce optimistic,
  untested instructions.
- **Example:** README says `make bootstrap`; that target was renamed
  three months ago.
- **Sources:** framework heuristic; informed by team retrospectives.

## Dimension 3.11 — Localization & internationalization

### AI-3.11-001 — Hardcoded user-visible strings

- **Status:** common knowledge.
- **Description:** Text shown to users is written directly in views
  instead of going through `String(localized:)` or a string catalog.
- **AI manifestation:** AI writes `Text("Save")` because it is the
  shortest correct expression for the prompt.
- **Example:** `Text("Continue")` instead of
  `Text(String(localized: "common.continue"))`.
- **Sources:** Apple Localization documentation.

### AI-3.11-002 — Locale-naive formatting

- **Status:** common knowledge.
- **Description:** Numbers, dates and currencies are formatted by
  string interpolation or by a formatter without an explicit locale,
  producing wrong output for non-default locales.
- **AI manifestation:** AI writes `"\(price) €"` or
  `NumberFormatter()` without setting `locale`.
- **Example:** `Text("$\(price)")` for a price displayed to a French
  user.
- **Sources:** Apple Localization documentation.

## Dimension 3.12 — UX & UI quality

### AI-3.12-001 — Computed-property views replacing real subviews

- **Status:** cited.
- **Description:** A SwiftUI screen is composed entirely of computed
  properties (`var header: some View { ... }`) instead of small
  reusable view types, leading to rebuild storms and poor
  composability.
- **AI manifestation:** Computed properties are the lowest-friction
  way for AI to "split a view"; it produces them by default.
- **Example:** A `HomeView` with eight `var someSection: some View`
  computed properties and no real subview type.
- **Sources:** Paul Hudson — *What to fix in AI-generated Swift code*
  (`hudson:281`).

### AI-3.12-002 — HIG-incoherent navigation

- **Status:** common knowledge.
- **Description:** Mixed navigation paradigms (push + sheet + modal +
  full-screen cover) used inconsistently across screens; back gestures
  break; navigation bars hide and reappear.
- **AI manifestation:** AI matches the verb in the prompt ("show",
  "open", "present") to whichever modifier sounds closest, without
  preserving the app's existing navigation grammar.
- **Example:** Some details push, others present a sheet, with no
  rule explaining when.
- **Sources:** Apple Human Interface Guidelines — Navigation, Modality.

## Dimension 3.13 — Accessibility

### AI-3.13-001 — Missing accessibility labels on icon-only controls

- **Status:** common knowledge.
- **Description:** Buttons whose content is an SF Symbol have no
  accessibility label, leaving VoiceOver users with "Button" as the
  only hint.
- **AI manifestation:** AI generates `Button { ... } label: { Image(systemName: "trash") }`
  without `.accessibilityLabel`.
- **Example:** A delete icon button with no accessible name.
- **Sources:** Apple Accessibility documentation; Paul Hudson on
  SwiftUI accessibility.

### AI-3.13-002 — Layouts that break under Dynamic Type

- **Status:** common knowledge.
- **Description:** Fixed heights, fixed widths, or single-line text
  truncate or clip when the user enables larger Dynamic Type sizes.
- **AI manifestation:** AI uses `.frame(height: 44)` for buttons or
  rows because it matches the "default" iOS height; the layout
  collapses at accessibility sizes.
- **Example:** A row with `.lineLimit(1)` and a fixed-height frame
  hiding the user's text on the largest Dynamic Type setting.
- **Sources:** Apple Accessibility documentation; Paul Hudson on
  Dynamic Type.

## Dimension 3.14 — Performance & energy

### AI-3.14-001 — Heavy work inside SwiftUI view bodies

- **Status:** common knowledge.
- **Description:** Computations, parsing or filtering of large
  collections happen inside `body`, re-running on every redraw.
- **AI manifestation:** AI inlines the transformation where the data
  is consumed, since that is the most direct expression of "show X
  filtered by Y".
- **Example:** `Text(items.filter { ... }.map { ... }.joined())`
  inside `body`.
- **Sources:** Apple SwiftUI performance documentation.

### AI-3.14-002 — Synchronous large-image decoding on the main thread

- **Status:** common knowledge.
- **Description:** Large images are loaded synchronously, decoded on
  the main thread, or used at full resolution where a thumbnail would
  suffice, causing scroll jank.
- **AI manifestation:** AI uses `Image(uiImage: UIImage(named:))`
  without considering size or thread; or decodes from `Data` on
  the main actor.
- **Example:** A list cell that loads a 4 MB image with
  `UIImage(contentsOfFile:)` on the main thread.
- **Sources:** Apple Image and SwiftUI performance documentation.

## Dimension 3.15 — API freshness & deprecations

### AI-3.15-001 — `NavigationView` instead of `NavigationStack`

- **Status:** cited.
- **Description:** New code uses `NavigationView`, which is
  deprecated, instead of `NavigationStack` / `NavigationSplitView`.
- **AI manifestation:** AI's training data still favors
  `NavigationView` for its breadth of examples.
- **Example:** A new screen scaffolded with `NavigationView { ... }`
  in an iOS 16+ project.
- **Sources:** Paul Hudson — *What to fix in AI-generated Swift code*
  (`hudson:281`).

### AI-3.15-002 — `ObservableObject` / `@StateObject` where `@Observable` is preferred

- **Status:** cited.
- **Description:** New view models adopt the `ObservableObject`
  protocol with `@Published` properties even though the project's
  deployment target supports `@Observable`.
- **AI manifestation:** Same training-data bias toward older patterns.
- **Example:** `final class HomeViewModel: ObservableObject` with
  `@Published var items: [Item] = []`, in an iOS 17+ codebase.
- **Sources:** Paul Hudson — *What to fix in AI-generated Swift code*
  (`hudson:281`); Apple Observation framework documentation.

### AI-3.15-003 — `.foregroundColor` instead of `.foregroundStyle`

- **Status:** cited.
- **Description:** New code uses `.foregroundColor`, which is
  deprecated, instead of `.foregroundStyle`.
- **AI manifestation:** Training-data bias.
- **Example:** `Text("Hello").foregroundColor(.red)` in a new file.
- **Sources:** Paul Hudson — *What to fix in AI-generated Swift code*
  (`hudson:281`).

### AI-3.15-004 — `tabItem`-based `TabView` API where the new `Tab` API applies

- **Status:** cited.
- **Description:** On iOS 18+, the new `Tab` / `TabView { Tab(...) }`
  API is preferred, but AI keeps producing the old
  `.tabItem { ... }` style.
- **AI manifestation:** Training-data bias.
- **Example:** `TabView { HomeView().tabItem { ... } }` in an iOS 18
  app instead of `TabView { Tab("Home", systemImage: "house") { ... } }`.
- **Sources:** Paul Hudson — *What to fix in AI-generated Swift code*
  (`hudson:281`).

## Dimension 3.16 — Code health & SOLID

### AI-3.16-001 — Narrative comments duplicating the code

- **Status:** common knowledge.
- **Description:** Comments restate what the next line obviously
  does, adding noise without context.
- **AI manifestation:** AI tends to comment generously, often
  explaining its own actions even when redundant.
- **Example:** `// Increment counter` immediately above `counter += 1`.
- **Sources:** common knowledge; reinforced by community style guides.

### AI-3.16-002 — God-files and growing-by-append edits

- **Status:** framework heuristic.
- **Description:** Files and types grow by append: each AI-assisted
  change adds a new method or property without refactoring,
  eventually producing a single file that owns most of a feature.
- **AI manifestation:** Without explicit instruction to refactor, AI
  prefers to add at the end of an existing file rather than introduce
  a new type.
- **Example:** A `HomeViewModel.swift` of 1500+ lines covering
  loading, persistence, formatting and analytics.
- **Sources:** framework heuristic.
