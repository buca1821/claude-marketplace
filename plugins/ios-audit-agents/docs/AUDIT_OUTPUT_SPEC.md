# Audit Output Specification

> **Spec v1.0.** Output contract for every audit run produced by the
> `ios-audit-agents` plugin: human-readable Markdown report and
> machine-readable JSON record.

This document is the contract under which agents and commands emit
their findings. It is referenced from the quality model
[`QUALITY_FRAMEWORK.md`](./QUALITY_FRAMEWORK.md) Section 4.

Every audit run **must** produce both:

- A Markdown report intended for humans (developers, leads, reviewers).
- A JSON record intended for machines (aggregation scripts, dashboards,
  longitudinal analysis).

Both files cover the same findings. The Markdown is a human projection
of the JSON.

---

## 1. File location and naming

Audit outputs live inside the audited repository, never in the plugin
itself or in user home folders, so that the audit history travels with
the project under version control if the team chooses.

### 1.1 Directory

```
<audited-repo>/.claude-marketplace-audits/
```

The directory is created by the plugin if missing. Teams may decide to
ignore it via `.gitignore` (private use) or commit it (auditable
history). The plugin does not enforce either.

### 1.2 File naming

Each audit run produces two files with the same base name:

```
<UTC-timestamp>__<audit-id>.md
<UTC-timestamp>__<audit-id>.json
```

- `<UTC-timestamp>`: ISO 8601 basic format without colons, UTC,
  e.g. `20260427T134205Z`.
- `<audit-id>`: short stable identifier of the run, format
  `[a-z0-9]{8}` (e.g. `9f3c1a72`). Generated by the plugin.

Example pair:

```
20260427T134205Z__9f3c1a72.md
20260427T134205Z__9f3c1a72.json
```

### 1.3 Optional index

When more than one audit exists in the directory, the plugin may
maintain an `index.json` file listing all runs in order. This is
optional and additive; it does not replace the per-run files.

## 2. Markdown report

The Markdown report is the human-readable projection of the audit. It
must be useful for a developer who has never seen the JSON.

### 2.1 Required structure

```markdown
# Audit report — <project name>

- Audit ID: <audit-id>
- Timestamp: <ISO 8601 UTC>
- Quality model version: <e.g. 0.1>
- Plugin version: <e.g. 1.3.2>
- Scope: <list of dimensions audited, e.g. 3.2, 3.12, 3.13, 3.14, 3.15, 3.16>
- Project commit: <git sha or "uncommitted">

## Executive summary

- Total findings: <N>
- By severity: P0 <a>, P1 <b>, P2 <c>, P3 <d>
- AI-typical: <X> of <N> (<ratio>%)
- Top recurring AI risk IDs: <top-3 if applicable>
- Highest severity items requiring attention: <count>

## Findings by dimension

### 3.X — <Dimension name>

#### <severity> — <finding title>

- ID: <auditor>-<short-slug>
- Severity: P0 | P1 | P2 | P3
- AI-typical: yes (<AI-3.X-NNN>) | no
- Evidence:
  - <file>:<line(s)>
  - <metric or log reference>
- Remediation: <what, who, by when>
- References: <backing reference(s)>

(repeat per finding)

(repeat per dimension that produced findings)

## Suggested remediation tasks

A flattened, severity-ordered list of remediation tasks, ready to be
copied into a tracker. Each entry includes the finding ID and a
one-line action.

- [P0] <task> (finding: <id>)
- [P1] <task> (finding: <id>)
- ...

## Methodology notes

- Agents that ran: <names>
- Skills consulted: <names>
- Dimensions in scope but with zero findings: <list>
- Dimensions out-of-plugin-scope: 3.1, 3.10
```

### 2.2 Style rules

- One report file per audit run. No appending to a previous file.
- The Markdown must be self-contained: a reader does not need to open
  the JSON to act on the findings.
- Code excerpts must follow the privacy policy in Section 4.
- Severity ordering inside a dimension: P0, P1, P2, P3.
- Dimension ordering: numerical (3.2 before 3.12).

## 3. JSON schema (v1.0)

The JSON record is the canonical data form. The Markdown is derived
from it. Aggregation, dashboards and longitudinal analysis consume
the JSON, never the Markdown.

### 3.1 Top-level shape

```json
{
  "schema_version": "1.0",
  "audit_id": "9f3c1a72",
  "timestamp": "2026-04-27T13:42:05Z",
  "model_version": "0.1",
  "plugin_version": "1.3.2",
  "project": {
    "name": "ExampleApp",
    "git_sha": "abc1234",
    "ios_deployment_target": "16.0"
  },
  "scope": {
    "dimensions_audited": ["3.2", "3.12", "3.15"],
    "agents_used": ["architecture-auditor", "ux-accessibility-auditor", "api-freshness-auditor"],
    "skills_used": ["framework", "ai-risk-catalog"]
  },
  "findings": [],
  "metrics": {
    "total_findings": 0,
    "by_severity": { "P0": 0, "P1": 0, "P2": 0, "P3": 0 },
    "by_dimension": {},
    "ai_typical_count": 0,
    "ai_typical_ratio": 0.0,
    "duration_seconds": 0
  },
  "notes": {
    "dimensions_in_scope_with_zero_findings": [],
    "dimensions_out_of_plugin_scope": ["3.1", "3.10"]
  }
}
```

### 3.2 Field reference

| Field | Type | Required | Description |
|---|---|---|---|
| `schema_version` | string | yes | Semver of this spec. v1.0 today. |
| `audit_id` | string | yes | `[a-z0-9]{8}`, stable per run. |
| `timestamp` | string | yes | ISO 8601 UTC with `Z` suffix. |
| `model_version` | string | yes | Version of the quality model (`QUALITY_FRAMEWORK.md`), e.g. `0.1`. |
| `plugin_version` | string | yes | Version from `plugin.json`. |
| `project.name` | string | yes | Project name. |
| `project.git_sha` | string | yes | Short or full SHA, or `"uncommitted"`. |
| `project.ios_deployment_target` | string | optional | e.g. `"16.0"`. |
| `scope.dimensions_audited` | string[] | yes | Dimension IDs as `"3.X"`. |
| `scope.agents_used` | string[] | yes | Agent names. |
| `scope.skills_used` | string[] | optional | Skill names. |
| `findings` | object[] | yes | See 3.3. May be empty. |
| `metrics.total_findings` | integer | yes | Count of `findings`. |
| `metrics.by_severity` | object | yes | Counts per `P0`/`P1`/`P2`/`P3`. |
| `metrics.by_dimension` | object | yes | Map dimension → count. |
| `metrics.ai_typical_count` | integer | yes | Count of `ai_typical: true`. |
| `metrics.ai_typical_ratio` | number | yes | `ai_typical_count / total_findings`, 0.0 if no findings. |
| `metrics.duration_seconds` | integer | yes | Wall-clock duration of the run. |
| `notes.dimensions_in_scope_with_zero_findings` | string[] | yes | Dimensions audited but producing no findings. Useful for detecting weak agents. |
| `notes.dimensions_out_of_plugin_scope` | string[] | yes | Always `["3.1", "3.10"]` in v0.1 of the quality model. |

### 3.3 Finding object

```json
{
  "id": "arch-001",
  "dimension": "3.15",
  "severity": "P2",
  "title": "NavigationView used in 4 views",
  "evidence": {
    "files": [
      { "path": "Views/Home.swift", "lines": [42, 87] },
      { "path": "Views/Detail.swift", "lines": [15] }
    ],
    "metrics": [],
    "logs": []
  },
  "ai_typical": true,
  "ai_risk_id": "AI-3.15-001",
  "remediation": "Migrate NavigationView usages to NavigationStack.",
  "references": ["hudson:281"]
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | yes | Stable per run, format `<auditor-prefix>-<slug>`. |
| `dimension` | string | yes | `"3.X"` from the quality model. |
| `severity` | enum | yes | `"P0" \| "P1" \| "P2" \| "P3"`. |
| `title` | string | yes | One-line human title. |
| `evidence.files` | object[] | optional | `{ path, lines[] }`. |
| `evidence.metrics` | object[] | optional | Free-form metric refs. |
| `evidence.logs` | object[] | optional | Free-form log refs. |
| `ai_typical` | boolean | yes | Whether this matches a catalog entry. |
| `ai_risk_id` | string | required if `ai_typical: true` | Stable ID `AI-3.X-NNN` from `AI_RISK_CATALOG.md`. |
| `remediation` | string | yes | Action-oriented sentence. |
| `references` | string[] | yes | Backing references. See 3.4. |

### 3.4 Reference identifiers

To keep references stable across audits, this spec defines a short
identifier scheme used in the `references` array:

| Prefix | Source |
|---|---|
| `apple:hig` | Apple Human Interface Guidelines |
| `apple:review` | Apple App Review Guidelines |
| `apple:a11y` | Apple Accessibility documentation |
| `apple:concurrency` | Apple Swift Concurrency documentation |
| `apple:testing` | Apple Swift Testing documentation |
| `apple:oslog` | Apple OSLog documentation |
| `apple:metrickit` | Apple MetricKit documentation |
| `apple:localization` | Apple Localization documentation |
| `masvs:<group>` | OWASP MASVS v2.1 + control group (e.g. `masvs:storage`) |
| `iso:25010` | ISO/IEC 25010:2023 |
| `wcag:2.2` | WCAG 2.2 |
| `hudson:<art>` | Paul Hudson, Hacking with Swift article number |
| `sundell:<topic>` | Swift by Sundell, topic slug |
| `wals:<topic>` | Donny Wals, topic slug |
| `vanderlee:<topic>` | Antoine van der Lee, topic slug |

References that do not fit this list use a free string and may be
normalized in a future spec version.

## 4. Privacy policy

The JSON is designed so it can be safely committed to the repository
or shared in aggregation. The following rules are non-negotiable.

### 4.1 Never include in evidence

- **Verbatim source code excerpts** longer than what is strictly
  needed to identify a finding. Use `path:line` references instead of
  pasting code.
- **Secrets in clear text**. When a secret is detected, the finding
  records its location and type, never its value.
- **Personally identifiable information** from commits (author email,
  full names, full commit messages). Use only the SHA.
- **Customer data** of any kind, including data found in fixtures or
  test files when its origin is real users.
- **Internal URLs, internal hostnames, or environment variables**
  whose disclosure would reveal infrastructure.

### 4.2 Recommended for evidence

- File paths relative to the repository root.
- Line numbers and ranges.
- Symbol names (function, class, type) when needed for clarity.
- Metric values (without identifying users).
- Log levels and categories (without log content).

### 4.3 Redaction expectations

When in doubt, the agent producing the finding **must redact**.
Redaction takes the form of a placeholder, e.g.:

```
"path": "Sources/Auth/Token.swift",
"lines": [42],
```

never:

```
"snippet": "let apiKey = \"sk_live_abc123...\""
```

---

## 5. Schema versioning

This spec follows semantic versioning at the schema level:

- **Major (`2.0`, `3.0`, ...)**: breaking change to the JSON shape
  (renamed/removed required fields, type changes). Old logs cannot be
  read by new tooling without a migration.
- **Minor (`1.1`, `1.2`, ...)**: backward-compatible additions
  (new optional fields, new enum values for non-required fields).
- **Patch**: clarification only, no field changes.

Tooling that consumes the JSON must check `schema_version` and refuse
to operate on a major newer than what it understands.

---

## 6. Aggregation hints

The JSON is shaped so that simple aggregation is possible without
custom infrastructure. The following questions can be answered by
iterating over JSON files in `.claude-marketplace-audits/`:

- *Which AI-typical risks recur the most?*
  Count `ai_risk_id` across all findings with `ai_typical: true`.
- *Which dimensions are silent?*
  Compare `scope.dimensions_audited` with `metrics.by_dimension`;
  silent dimensions appear in the former but not the latter, or are
  listed under `notes.dimensions_in_scope_with_zero_findings`.
- *Is severity drifting over time?*
  Sort runs by `timestamp`; track `metrics.by_severity` series.
- *Is `ai_typical_ratio` rising or falling?*
  Track `metrics.ai_typical_ratio` series.

These are the inputs that feed Section 7.3 of the quality model.
